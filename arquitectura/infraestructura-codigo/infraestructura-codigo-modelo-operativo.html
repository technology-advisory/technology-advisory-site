Operaciones · Infraestructura como Código

Modelo operativo: pipelines que aplican cambios sin que nadie tenga que acordarse

Si aplicar el código de infraestructura depende de que alguien ejecute un comando manualmente, tienes IaC de nombre pero no de práctica.

Miguel Ángel Carriazo Infraestructura como Código Lectura Avanzada (14-18 min)
Módulo 01 — El Problema Central

El comando manual que rompe toda la disciplina

Tener el código en Git no basta si alguien sigue ejecutando terraform apply manualmente desde su portátil. Ese comando manual reintroduce exactamente el problema que IaC pretendía resolver: nadie más sabe qué se aplicó, cuándo, ni con qué versión del código.

El comando manual es el talón de Aquiles de cualquier estrategia de IaC. Por muy bonito que sea el código, por muy bien que esté versionado, si el momento de aplicar el cambio sigue siendo manual, el proceso se rompe. El apply manual es el punto donde la disciplina desaparece y el riesgo entra.

¿Por qué es un problema el apply manual? Porque:

  • No deja trazabilidad: no hay registro de quién aplicó el cambio
  • No garantiza consistencia: el código que se aplica no es necesariamente el de la rama principal
  • Permite errores: un apply manual puede ejecutarse con variables incorrectas
  • Rompe el proceso de revisión: la aprobación en el PR se vuelve irrelevante
⚠️ El apply manual es el síntoma de un Nivel 2: código en Git, pero sin proceso alrededor. La madurez real empieza cuando el apply solo ocurre a través del pipeline.

La solución no es "confiar en que nadie hará apply manual" — es hacer que sea imposible hacer apply manual. Si el pipeline es el único camino para aplicar cambios, el proceso se cumple por defecto, sin depender de la disciplina individual.

⚙ Módulo 02
Módulo 02 — El Pipeline

Pipeline con plan y apply separados

Un pipeline maduro de IaC separa el plan (calcular qué cambiaría, sin aplicar nada) del apply (ejecutar el cambio). El plan se ejecuta automáticamente en cada pull request, mostrando el diff exacto; el apply solo ocurre tras aprobación explícita y fusión a la rama principal.

Esta separación es fundamental. El plan es seguro: puedes ejecutarlo tantas veces como quieras sin miedo a romper nada. Te muestra exactamente qué va a cambiar, te permite revisarlo, discutirlo y aprobarlo. Solo cuando el plan ha sido revisado y aprobado, se ejecuta el apply, que sí hace cambios.

La separación plan/apply también permite auditar el proceso. El historial del pipeline muestra qué cambios se planificaron, quién los aprobó, y cuándo se aplicaron. Es un registro de auditoría completo que no depende de que nadie recuerde haber ejecutado un comando.

Práctica recomendada: el pipeline ejecuta terraform plan en cada PR y publica el diff como comentario. El apply solo se ejecuta al hacer merge a la rama principal, y solo después de aprobación explícita.

Herramientas como Atlantis, Terraform Cloud, o GitHub Actions con Terraform permiten implementar esta separación de forma sencilla. La elección de la herramienta es menos importante que el principio: plan en PR, apply en merge.

⚙ Módulo 03
Módulo 03 — El Juicio Humano

Aprobación humana en el punto correcto del proceso

La automatización de IaC no elimina la necesidad de juicio humano — la reubica en el momento correcto: antes de aplicar, revisando el plan generado, no después, descubriendo por accidente qué cambió. Para cambios en producción, una aprobación explícita de una segunda persona (no el autor del cambio) es la práctica defendible.

El juicio humano en el pipeline de IaC se centra en lo que la automatización no puede hacer:

  • Evaluar el impacto operativo del cambio: ¿requiere mantenimiento? ¿Afecta a otros sistemas?
  • Considerar el contexto temporal: ¿es seguro aplicar esto ahora o debería esperar a una ventana de mantenimiento?
  • Detectar intenciones implícitas: el código dice una cosa, pero el revisor puede detectar matices que el plan no muestra
  • Validar que la documentación del cambio es adecuada: ¿está el ticket actualizado? ¿Hay un plan de rollback?
Práctica recomendada: para cambios en producción, requiere aprobación explícita de una persona distinta al autor del PR. El "cuatro ojos" es el estándar defendible para infraestructura crítica.

La aprobación humana no es una barrera — es un seguro. La automatización da velocidad; el juicio humano da seguridad. La combinación de ambos es lo que permite operar infraestructura crítica con confianza.

⚙ Módulo 04
Módulo 04 — El Principio DRY

Ambientes: dev, staging, producción con el mismo código

El mismo módulo de Terraform, parametrizado por ambiente (variables distintas, no código distinto), es lo que garantiza que lo que se probó en staging se comporte igual en producción. Mantener código duplicado y ligeramente distinto por ambiente reintroduce el riesgo de divergencia que IaC pretende eliminar.

El principio de "Don't Repeat Yourself" (DRY) es fundamental en IaC. Si tienes un módulo para dev y otro casi idéntico para producción, estás duplicando código y, lo que es peor, duplicando el riesgo. Un cambio que se hace en dev pero no en producción, o viceversa, es una divergencia que puede causar problemas impredecibles.

La estructura recomendada para ambientes en IaC es:

  • Un solo módulo para el recurso o conjunto de recursos
  • Variables de ambiente (terraform.tfvars o variables de pipeline) para configuraciones específicas
  • Workspaces o directorios separados para cada ambiente, pero usando el mismo código
⚠️ Antipatrón: código duplicado y ligeramente distinto por ambiente. La divergencia entre ambientes es la causa principal de incidentes en producción: lo que no se probó en staging no debería ir a producción.

La parametrización por ambiente no significa que todos los ambientes sean iguales. Un ambiente de desarrollo puede tener instancias más pequeñas, menos réplicas, o almacenamiento reducido. Pero la estructura del código — los recursos que se crean, sus relaciones, sus configuraciones de seguridad — debe ser idéntica en todos los ambientes. La diferencia está en los valores de las variables, no en la lógica del código.

⚙ Módulo 05
Módulo 05 — El Estándar

De pipeline artesanal a estándar de equipo

El pipeline que funciona para un proyecto piloto necesita convertirse en plantilla reutilizable para el resto del equipo — módulos compartidos, convenciones de nombrado, estructura de repositorio estándar — antes de escalar a decenas de proyectos con configuraciones inconsistentes entre sí.

El camino desde un pipeline artesanal (hecho para un proyecto específico) a un estándar de equipo es:

  • Documentar el pipeline: qué hace, cómo se usa, qué requiere
  • Extraer módulos comunes: recursos que se repiten en varios proyectos
  • Estandarizar variables: nombres de variables, estructura de tfvars
  • Crear plantillas de repositorio para nuevos proyectos
  • Establecer convenciones de nombrado, tagging, y estructura de archivos

Un estándar de equipo no es burocracia — es la diferencia entre un equipo que escala y uno que se fragmenta. Cuando cada proyecto tiene su propia forma de hacer las cosas, el conocimiento no se comparte y la infraestructura se vuelve inconsistente. El estándar permite que los miembros del equipo se muevan entre proyectos sin tener que aprender configuraciones completamente nuevas.

✔ La señal de madurez operativa: un nuevo miembro del equipo puede crear un nuevo proyecto siguiendo la plantilla estándar, y el pipeline funciona sin configuración adicional. Eso es estándar de equipo.

El estándar de equipo en IaC también facilita la auditoría y el compliance. Cuando todos los proyectos siguen la misma estructura, las herramientas de escaneo y las políticas como código pueden aplicarse de forma consistente. Un estándar no es una restricción — es una habilitación: libera al equipo de tener que resolver los mismos problemas una y otra vez.
