Controles y evidencias: el drift que nadie detectó durante meses

Qué evidencias de IaC reunir, y por qué la más reveladora es la comparación entre lo que el código dice y lo que realmente existe en producción.

Miguel Ángel Carriazo Infraestructura como Código Lectura Avanzada (16-20 min)
Módulo 01 — El Síntoma Silencioso

El drift que nadie detectó durante meses

El drift ocurre cuando alguien modifica manualmente un recurso que está gestionado por Terraform, sin pasar por el código. El código sigue diciendo una cosa; la realidad en producción dice otra. Sin detección activa de drift, esa discrepancia puede persistir meses hasta que un terraform apply intenta "corregir" el cambio manual, con resultados inesperados.

El drift es el equivalente en infraestructura de la deuda técnica: no duele hasta que duele. Puedes tener meses de drift sin darte cuenta, hasta que un día alguien ejecuta un apply y descubre que el firewall que creías que estaba abierto en el código fue cerrado manualmente hace semanas — y nadie se acuerda por qué.

La detección activa de drift es la única forma de evitar que estas discrepancias se conviertan en incidentes. No es un lujo de organizaciones grandes; es un control básico para cualquier equipo que use IaC.

⚠️ El drift es la brecha silenciosa: el código dice una cosa, la realidad dice otra. Sin detección automática, esa brecha puede persistir meses.
⚙ Módulo 02
Módulo 02 — El Registro de Auditoría

Historial de Git como evidencia de cambios

El historial de commits de un repositorio de Terraform es, en sí mismo, un registro de auditoría de cambios de infraestructura: quién propuso el cambio, quién lo aprobó, cuándo se aplicó. Esta evidencia, generada como subproducto natural del flujo de trabajo, es mucho más completa que cualquier registro manual paralelo.

Cada commit en el repositorio de IaC debería estar vinculado a un ticket de cambios (Jira, ServiceNow, etc.) que documente el motivo del cambio. Esta trazabilidad es lo que permite a un auditor reconstruir por qué se hizo un cambio determinado, sin tener que confiar en la memoria de nadie.

La práctica de "commits atómicos" — cada commit representa un cambio lógico completo y autónomo — es especialmente importante en IaC. Hace que el historial sea navegable, que los rollbacks sean seguros, y que el equipo pueda entender la evolución de la infraestructura sin tener que descifrar commits gigantes.

Práctica recomendada: todos los cambios de infraestructura deben pasar por pull request con aprobación explícita. El merge a la rama principal es el evento que desencadena el pipeline de aplicación.
⚙ Módulo 03
Módulo 03 — Seguridad Integrada

Escaneo de seguridad en el pipeline, no después del despliegue

Herramientas como tfsec o Checkov analizan el código de Terraform antes de aplicarlo, detectando configuraciones inseguras (buckets S3 públicos, security groups demasiado abiertos, cifrado no habilitado) en el momento del pull request, no después de que el recurso inseguro ya esté en producción expuesto.

La diferencia entre escanear antes o después del despliegue es enorme. Escanear después es reactivo: el recurso ya está en producción, el daño potencial ya existe. Escanear antes es preventivo: el recurso nunca llega a producción en un estado inseguro. Es la diferencia entre "arreglaremos esto cuando podamos" y "esto no se despliega así".

Práctica recomendada: el escaneo de seguridad como gate obligatorio del pipeline — un hallazgo crítico bloquea el merge, no genera solo una alerta que se puede ignorar.

Las herramientas de escaneo estático son el primer filtro, pero no el único. La combinación de tfsec/Checkov (para reglas predefinidas) y OPA/Sentinel (para políticas personalizadas de la organización) proporciona una cobertura completa que detecta tanto problemas comunes como requisitos específicos de compliance.

⚙ Módulo 04
Módulo 04 — Prevención Proactiva

Detectar drift antes de que sea un problema

Ejecutar terraform plan periódicamente contra el estado actual (sin aplicar nada) revela cualquier drift acumulado — diferencias entre lo que el código declara y lo que realmente existe. Programar esta verificación como tarea automática diaria o semanal, con alerta si se detecta drift, es la práctica que evita sorpresas.

El drift es inevitable en entornos donde coexisten cambios manuales y automatización. Pero la diferencia entre una organización madura y una inmadura no es la ausencia de drift — es cómo lo detecta y gestiona. La detección automática convierte el drift en una excepción gestionable, no en una bomba de relojería.

La práctica recomendada es:

  • Programar un terraform plan -refresh diario en cada ambiente
  • Enviar el resultado a un sistema de logs o dashboard
  • Alertar si el plan detecta cambios no declarados
  • Establecer un procedimiento para resolver drift: o actualizar el código o revertir el cambio manual
⚠️ El drift no es un error — es una señal: indica que alguien ha hecho un cambio fuera del proceso. La señal es valiosa, pero solo si se detecta y se actúa sobre ella.
⚙ Módulo 05
Módulo 05 — La Auditoría de IaC

La evidencia que un auditor de IaC pide de verdad

Más allá del código en sí, la evidencia de madurez incluye: historial de aprobaciones de cambios en producción, resultados de escaneos de seguridad por cada cambio, y registro de detecciones de drift con su resolución. Sin estos tres elementos, tener el código en Git es solo el primer paso, no la evidencia completa de un proceso maduro.

Un auditor de IaC no se conforma con ver el repositorio. Pregunta:

  • ¿Todos los cambios de infraestructura pasan por pull request?
  • ¿Hay evidencia de que se revisa el plan antes de aplicar?
  • ¿Cómo se detecta y resuelve el drift?
  • ¿Los escaneos de seguridad son obligatorios en el pipeline?

La madurez en IaC se demuestra con evidencia, no con intención. Un repositorio con código pero sin proceso alrededor es solo código — no es gobernanza.

✔ Las tres evidencias de madurez: historial de Git + escaneos de seguridad + detección de drift. Sin estas tres, el código es solo un artefacto, no una práctica.
