Principios de diseño: el clic en la consola que nadie puede reproducir

Cada cambio hecho a mano en una consola cloud o en un vCenter es un cambio que nadie puede versionar, revisar ni reproducir de forma fiable. IaC no es una moda de DevOps, es la diferencia entre infraestructura auditable e infraestructura de fe.

Miguel Ángel Carriazo Infraestructura como Código Lectura Avanzada (18-22 min)
Módulo 01 — El Problema de Fondo

La infraestructura de fe

"Funciona, no lo toques" es la frase que revela infraestructura sin código: nadie sabe exactamente por qué funciona, porque no hay un documento vivo que lo explique — solo el estado actual, alcanzado por una sucesión de clics que nadie registró.

La infraestructura de fe es aquella que depende del conocimiento tácito de una o dos personas. Funciona mientras esas personas están disponibles, pero se convierte en un problema cuando se van de vacaciones, cambian de rol o simplemente olvidan cómo hicieron aquel cambio hace tres años. La infraestructura no debería depender de la memoria humana — debería depender de código que pueda leerse, revisarse y ejecutarse.

El primer principio de IaC es: **todo cambio de infraestructura debe pasar por código**. No hay excepciones para "arreglos rápidos" o "cambios temporales". Cada clic en una consola es un cambio que no puede ser versionado, revisado ni reproducido. Y cada cambio no versionado es deuda técnica que alguien pagará más tarde.

⚠️ Riesgo real: el único administrador que sabe cómo está configurado un sistema crítico se va de vacaciones, o de la empresa, y ese conocimiento se va con él. La infraestructura de fe es insostenible.

La transición de infraestructura de fe a infraestructura como código no es solo técnica — es cultural. Requiere que los equipos confíen en que el código puede describir su infraestructura, y que el proceso de revisión y aplicación es tan fiable como (o más que) el clic manual. Es un cambio de mentalidad que lleva tiempo, pero es el único camino hacia una infraestructura sostenible.

⚙ Módulo 02
Módulo 02 — El Enfoque Correcto

Declarativo vs imperativo: la diferencia que importa

Terraform y herramientas similares son declarativas: describes el estado final deseado, y la herramienta calcula qué cambios aplicar para llegar ahí. Un script imperativo (una secuencia de comandos "haz esto, luego esto") describe pasos, no el resultado — y falla de forma distinta si algo ya existe parcialmente o el orden se altera.

La diferencia entre declarativo e imperativo es fundamental para entender IaC. Un script imperativo dice: "crea una VM, luego instala Apache, luego inicia el servicio". Si la VM ya existe, el script puede fallar o crear una duplicada. Si el servicio ya está iniciado, el script puede intentar iniciarlo de nuevo y producir un error.

Un enfoque declarativo dice: "quiero una VM con Apache corriendo". La herramienta verifica el estado actual, calcula la diferencia, y aplica solo los cambios necesarios para llegar al estado deseado. Si ya existe una VM con Apache corriendo, no hace nada. Si falta Apache, lo instala. Si el servicio no está iniciado, lo inicia. Es idempotente: ejecutarlo múltiples veces produce el mismo resultado.

Esta idempotencia es lo que hace que IaC sea fiable y segura. Puedes ejecutar un plan tantas veces como quieras sin miedo a romper cosas, porque solo aplica los cambios necesarios. Y puedes revisar el plan antes de aplicarlo, entendiendo exactamente qué va a cambiar.

Consejo práctico: si estás escribiendo scripts imperativos para gestionar infraestructura, pregúntate si una herramienta declarativa (Terraform, CloudFormation, Pulumi) podría hacer el trabajo de forma más fiable y mantenible. La respuesta casi siempre es "sí".

La herramienta declarativa adecuada depende del contexto: Terraform para multi-cloud y on-premise, CloudFormation para AWS nativo, ARM para Azure, Pulumi para equipos que prefieren lenguajes de programación generales. Pero el principio es el mismo: describes el estado deseado, no los pasos para llegar a él.

⚙ Módulo 03
Módulo 03 — El Mapa entre Código y Realidad

El estado: la pieza que todos rompen alguna vez

El fichero de estado de Terraform es el mapa entre tu código y los recursos reales desplegados. Si se corrompe, se pierde, o dos personas lo modifican simultáneamente sin bloqueo (state locking), el resultado puede ser un desastre: Terraform intentando recrear recursos que ya existen, o eliminando recursos que sigue creyendo huérfanos.

El estado es el activo más importante de cualquier proyecto de Terraform. Sin él, la herramienta no sabe qué gestiona. Con él corrompido, la herramienta hace cosas inesperadas. La gestión del estado es tan crítica como la escritura del código, pero a menudo se descuida hasta que ocurre un problema.

Las prácticas recomendadas para el estado son claras:

  • Backend remoto: S3 + DynamoDB (AWS), Azure Storage + Blob Lock (Azure), GCS (GCP), o Terraform Cloud
  • State locking: impide que dos personas modifiquen el estado simultáneamente
  • Respaldos automáticos: la mayoría de backends remotos los ofrecen por defecto
  • Nunca editar el estado manualmente: siempre a través de terraform apply o terraform import
⚠️ Configuración obligatoria: backend remoto (S3, Azure Storage, Terraform Cloud) con state locking activado. El estado local en el portátil de alguien no es una opción viable para equipos. Si trabajas solo, sigue siendo una mala práctica — el estado local no tiene respaldo.

El error más común con el estado es intentar arreglarlo manualmente cuando algo va mal. Si el estado está corrupto, no lo edites a mano — usa terraform state commands, o si es necesario, reconstruye desde cero. Editar el estado manualmente casi siempre empeora las cosas.

⚙ Módulo 04
Módulo 04 — El Proceso de Revisión

Revisión de código para infraestructura, no solo para aplicaciones

Un cambio de infraestructura vía Terraform debería pasar por el mismo proceso de pull request y revisión que un cambio de código de aplicación. El terraform plan generado se adjunta al PR, mostrando exactamente qué se va a crear, modificar o destruir — visibilidad que un clic manual en una consola nunca ofrece.

La revisión de código de infraestructura es diferente a la revisión de código de aplicación. No solo se revisa la sintaxis o la lógica — se revisa el impacto operativo: ¿este cambio afecta a la disponibilidad? ¿Requiere mantenimiento programado? ¿Crea una dependencia no documentada? ¿Es seguro hacerlo a mediodía o debería hacerse en ventana de mantenimiento?

Un pull request de infraestructura bien hecho incluye:

  • El terraform plan completo, con diff visible
  • Una descripción del cambio y su motivo
  • El ticket de cambios asociado (Jira, ServiceNow, etc.)
  • La evidencia de que el cambio se ha probado en staging
  • El plan de rollback si algo sale mal
Práctica recomendada: el plan generado por Terraform se publica automáticamente como comentario en el PR. El revisor no tiene que ejecutar nada localmente — todo está visible en el propio PR.

La revisión de infraestructura no es solo para detectar errores — es para compartir conocimiento. Cada PR revisado es una oportunidad para que el equipo aprenda sobre la infraestructura, entienda las dependencias, y desarrolle un entendimiento compartido de cómo funciona el sistema. Es el mecanismo que convierte el conocimiento tácito en conocimiento explícito.

⚙ Módulo 05
Módulo 05 — El Camino Práctico

Migrar sin romper lo que ya funciona

Migrar infraestructura existente (creada a mano) a IaC no significa recrearla desde cero. Herramientas como terraform import permiten traer recursos existentes bajo gestión de Terraform sin destruirlos y recrearlos — el camino defendible para no arriesgar producción en el proceso de adoptar buenas prácticas.

El proceso de migración a IaC para infraestructura existente es:

  • Auditoría: documenta qué recursos existen actualmente
  • Escribir código: define los recursos en Terraform basándote en la auditoría
  • Importar: usa terraform import para traer recursos existentes al estado
  • Verificar: ejecuta terraform plan para confirmar que no hay diferencias
  • Validar: prueba en un entorno no crítico antes de producción

El import no es perfecto — a veces hay diferencias entre lo que el código declara y lo que realmente existe, especialmente en recursos creados hace tiempo. Esas diferencias se resuelven ajustando el código o, en algunos casos, haciendo pequeños cambios de reconciliación.

✔ El camino defendible: importar recursos existentes antes de hacer cambios. Esto te permite poner infraestructura existente bajo gestión de IaC sin riesgo de eliminar nada. Una vez importada, los cambios futuros pasan por el código.

La migración a IaC es un proceso gradual. No necesitas importar toda la infraestructura el primer día. Empieza por los recursos menos críticos, aprende con ellos, y extiende el proceso a los recursos más críticos cuando tengas confianza. Es más lento, pero es la diferencia entre una migración exitosa y un incidente evitable.
