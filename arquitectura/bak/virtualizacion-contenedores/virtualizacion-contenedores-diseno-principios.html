Principios de diseño: un contenedor no es una VM ligera, es otro modelo de seguridad

Tratar los contenedores como "VMs pequeñas" es el error conceptual que produce despliegues inseguros. Comparten kernel con el host — el aislamiento es de proceso, no de hardware virtualizado.

Miguel Ángel Carriazo Virtualización y Contenedores Lectura Avanzada (18-22 min)
Módulo 01 — El Error Conceptual

"VM ligera" es un error conceptual con consecuencias reales

Un contenedor no es una máquina virtual más pequeña y rápida. Una VM tiene su propio kernel, aislado por el hipervisor a nivel de hardware virtualizado. Un contenedor comparte el kernel del host — el aislamiento es lógico (namespaces, cgroups), no físico. Esa diferencia cambia completamente el modelo de amenazas.

Este error conceptual es común porque los contenedores parecen VMs: tienen su propio sistema de ficheros, su propia red, sus propios procesos. Pero debajo del capó, la arquitectura es completamente diferente. Una VM virtualiza hardware; un contenedor virtualiza el sistema operativo. La diferencia en el modelo de seguridad es enorme.

Las implicaciones de seguridad de esta diferencia son:

  • Un escape de contenedor puede comprometer el kernel del host
  • Un contenedor privilegiado tiene acceso a recursos del host que una VM no tendría
  • Las vulnerabilidades del kernel afectan a todos los contenedores del host
⚠️ Consecuencia directa: una vulnerabilidad de escape de contenedor puede comprometer el kernel del host y, con él, todos los demás contenedores que corren sobre esa misma máquina. En una VM, el hipervisor aísla el kernel de la VM del host.
⚙ Módulo 02
Módulo 02 — Mecanismos de Aislamiento

Namespaces y cgroups: cómo aísla de verdad un contenedor

Los namespaces de Linux dan a cada contenedor su propia vista aislada de procesos, red, punto de montaje e identificadores de usuario. Los cgroups limitan y contabilizan el uso de recursos (CPU, memoria, I/O). Juntos crean la ilusión de aislamiento — pero el kernel subyacente es uno solo, compartido por todos los contenedores del host.

Los principales namespaces que usa Docker/Kubernetes son:

  • PID: los procesos dentro del contenedor ven su propio espacio de IDs
  • NET: cada contenedor tiene su propia interfaz de red y tabla de rutas
  • MNT: sistema de ficheros aislado dentro del contenedor
  • USER: mapeo de usuarios — el root dentro del contenedor no es root en el host
  • IPC: comunicación entre procesos aislada

Los cgroups, por su parte, garantizan que un contenedor no pueda consumir todos los recursos del host, protegiendo a otros contenedores y al propio sistema operativo de denial of service. La combinación de namespaces y cgroups hace que los contenedores sean seguros por defecto — pero solo si se usan correctamente.

El aislamiento del contenedor es lógico, no físico: namespaces + cgroups crean la ilusión de aislamiento, pero el kernel es compartido. Esto es lo que permite la eficiencia de los contenedores y también lo que crea su modelo de amenazas específico.
⚙ Módulo 03
Módulo 03 — El Riesgo de Privilegios

Root en el contenedor puede significar root en el host

Ejecutar un contenedor con privilegios elevados (--privileged en Docker, o un Pod sin restricciones en Kubernetes) reduce drásticamente el aislamiento — un proceso con privilegios dentro del contenedor puede, en muchos escenarios, acceder a recursos del host que un contenedor bien configurado nunca debería alcanzar.

El flag --privileged da a un contenedor acceso a todos los dispositivos del host y elimina la mayoría de las restricciones del kernel. En Kubernetes, un Pod sin restricciones (especialmente en modo Privileged) puede hacer lo mismo. Esto es equivalente a ejecutar el proceso como root en el host — y es una práctica que nunca debería ser el default.

La configuración defendible por defecto es:

  • Usuario no-root dentro del contenedor (definido en el Dockerfile)
  • Capabilities mínimas (drop todas, añadir solo las necesarias)
  • Sistema de ficheros de solo lectura donde sea posible
  • Sin flag --privileged salvo excepción justificada
⚠️ Root en el contenedor no es root en el host, gracias al user namespace — pero la diferencia es mucho menor en un contenedor privilegiado. El user namespace mapea el root del contenedor a un usuario no privilegiado en el host, pero --privileged desactiva esa protección.
⚙ Módulo 04
Módulo 04 — La Decisión de Arquitectura

Cuándo elegir VM, cuándo contenedor

La decisión no es "contenedores porque están de moda" — es evaluar el nivel de aislamiento que la carga de trabajo necesita. Multi-tenancy con clientes que no confían entre sí, o cargas con requisitos regulatorios estrictos de aislamiento, siguen justificando VMs. Microservicios internos de una misma organización, con confianza razonable entre ellos, se benefician del contenedor.

La pregunta clave para decidir entre VM y contenedor es: ¿Qué nivel de aislamiento necesita esta carga de trabajo?

  • Si el aislamiento es crítico (multi-tenant, clientes externos, regulación estricta) → VM
  • Si la velocidad y eficiencia importan más (microservicios internos, CI/CD) → contenedor
  • Si hay dudas → empieza por contenedores con políticas de seguridad estrictas, y si no es suficiente, pasa a VM
Consejo práctico: para la mayoría de las cargas internas, los contenedores con Pod Security Standards Baseline son suficientes. Para cargas multi-tenant sin confianza, las VMs siguen siendo la opción más segura.
⚙ Módulo 05
Módulo 05 — Seguridad por Defecto

Runtime seguro por defecto, no como añadido posterior

Las políticas de seguridad de contenedores (Pod Security Standards en Kubernetes, o equivalentes) deberían aplicarse por defecto a todo el clúster desde el primer día — usuario no-root, sistema de ficheros de solo lectura donde sea posible, sin privilegios elevados salvo excepción justificada — no añadirse retroactivamente cuando ya hay cientos de cargas desplegadas sin esas restricciones.

El enfoque de "seguridad por defecto" es el que permite escalar de forma segura. Si la seguridad es una excepción que hay que añadir después, la mayoría de las cargas no la tendrán. Si la seguridad es el default, la mayoría de las cargas la tendrán automáticamente, y las excepciones serán gestionadas como lo que son: excepciones.

La implementación práctica de seguridad por defecto en contenedores incluye:

  • Pod Security Standards activados a nivel de namespace (Baseline o Restricted)
  • Escaneo de imágenes obligatorio en el pipeline
  • RBAC con privilegios mínimos para ServiceAccounts
  • Políticas de red que limitan la comunicación entre pods
  • Auditoría continua del clúster
✔ El principio de seguridad por defecto: si tienes que añadir seguridad a cada despliegue, la seguridad es una opción. Si la seguridad viene por defecto y solo se desactiva con justificación, la seguridad es el estándar.
