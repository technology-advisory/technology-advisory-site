Modelo operativo: quién actualiza la imagen base, y con qué frecuencia

Una imagen de contenedor construida hace un año y nunca reconstruida acumula vulnerabilidades conocidas silenciosamente, aunque la aplicación en sí no haya cambiado.

Miguel Ángel Carriazo Virtualización y Contenedores Lectura Avanzada (14-18 min)
Módulo 01 — El Problema Silencioso

La imagen que nadie reconstruye

Una imagen de contenedor construida hace un año, sin reconstruir desde entonces, tiene la misma versión del sistema operativo base, las mismas librerías, y probablemente varias vulnerabilidades conocidas que no existían cuando se construyó. El código de la aplicación puede no haber cambiado; el riesgo de la imagen sí.

Este es uno de los problemas más comunes y menos visibles en la gestión de contenedores. La imagen se construye una vez, se despliega, y se olvida. Meses después, la imagen sigue en producción, pero el sistema operativo base y las librerías que contiene tienen vulnerabilidades que se han hecho públicas mientras tanto. La aplicación no ha cambiado, pero la imagen es insegura.

El problema es especialmente grave porque las vulnerabilidades de la imagen base son de alta severidad: afectan al sistema operativo, a librerías comunes como OpenSSL, a lenguajes de programación como Python o Node.js. Y la única forma de resolverlas es reconstruir la imagen con una base actualizada.

⚠️ El error más común: construir una imagen una vez y desplegarla durante meses o años sin reconstruir. Las vulnerabilidades de la imagen base se acumulan con el tiempo, aunque la aplicación no cambie. La imagen envejece en silencio.
⚙ Módulo 02
Módulo 02 — Seguridad en el Pipeline

Escaneo de vulnerabilidades antes de publicar, no después del incidente

Herramientas como Trivy, Grype o el escáner nativo de tu registro (Docker Hub, ACR, ECR) analizan cada imagen antes de publicarla, identificando CVEs conocidos en el sistema operativo base y las dependencias. Integrar este escaneo como gate obligatorio del pipeline —no como revisión ocasional— es lo que evita que vulnerabilidades conocidas lleguen a producción.

La diferencia entre escanear antes o después del despliegue es enorme. Escanear después es reactivo: la vulnerabilidad ya está en producción, y el equipo tiene que correr para parchear. Escanear antes es preventivo: la imagen nunca llega a producción con vulnerabilidades conocidas.

Los escáneres de imágenes modernos pueden:

  • Detectar vulnerabilidades en el sistema operativo base (Debian, Alpine, Ubuntu, etc.)
  • Detectar vulnerabilidades en dependencias de lenguajes (Python, Node.js, Java, etc.)
  • Generar reportes con severity (CRITICAL, HIGH, MEDIUM, LOW)
  • Bloquear automáticamente imágenes con vulnerabilidades CRITICAL o HIGH
Práctica recomendada: integra Trivy o Grype en tu pipeline CI/CD. Bloquea la publicación de imágenes que tengan vulnerabilidades CRITICAL. Para HIGH, evalúa si bloquear o solo alertar según el contexto. Nunca despliegues imágenes sin escaneo.
⚙ Módulo 03
Módulo 03 — Versionado de Imágenes

"Latest" no es una versión, es una trampa operativa

Usar el tag latest en producción significa que no sabes exactamente qué versión de la imagen está corriendo — puede cambiar sin que lo notes en el siguiente despliegue. Los tags inmutables (por hash de commit, por versión semántica) garantizan que lo que se prueba es exactamente lo que se despliega, sin sorpresas.

El problema de latest es que es un tag mutable. Puede apuntar a una imagen diferente en diferentes momentos. Cuando haces un docker pull de latest, obtienes la última versión, que puede ser diferente de la que desplegaste la semana pasada. Esto hace que los despliegues sean impredecibles y que los rollbacks sean casi imposibles.

Las prácticas recomendadas para tags en producción son:

  • Versión semántica: v1.2.3 para releases estables
  • Hash de commit: git-abc123 para builds de desarrollo
  • Fecha + hash: 2026-01-01-abc123 para builds diarios
  • Nunca usar "latest" en producción — solo en entornos de desarrollo
⚠️ Antipatrón habitual: despliegues en producción apuntando a latest, sin que nadie pueda decir con certeza qué versión exacta está corriendo en un momento dado. Esto hace que los rollbacks sean impredecibles y que la trazabilidad sea nula.
⚙ Módulo 04
Módulo 04 — Mantenimiento Programado

Reconstrucción periódica programada, aunque nada haya cambiado

Reconstruir imágenes periódicamente (mensualmente, como mínimo) desde una imagen base actualizada, aunque el código de la aplicación no haya cambiado, incorpora los parches de seguridad del sistema operativo y las librerías subyacentes. Sin esta disciplina, las imágenes envejecen en silencio.

La reconstrucción periódica es uno de los controles más infravalorados en la gestión de contenedores. No es glamuroso, no es técnicamente complejo, pero es la diferencia entre tener imágenes seguras y tener imágenes llenas de vulnerabilidades conocidas.

El proceso de reconstrucción periódica debería ser:

  • Frecuencia: mensual como mínimo, semanal si tu organización tiene capacidad
  • Base actualizada: usa siempre la última versión de la imagen base con parches de seguridad
  • Escaneo: después de reconstruir, escanea la nueva imagen para verificar que no hay nuevas vulnerabilidades
  • Despliegue: despliega la imagen reconstruida en los ambientes correspondientes
  • Verificación: valida que la aplicación sigue funcionando correctamente
Recomendación: automatiza la reconstrucción de imágenes con un pipeline programado (cron job en Jenkins, GitHub Actions schedule, etc.). La reconstrucción debe ocurrir sin intervención manual, como parte del mantenimiento rutinario.
⚙ Módulo 05
Módulo 05 — El Registro como Fuente de Verdad

Registro de imágenes como fuente de verdad auditable

Un registro de imágenes (container registry) con políticas de retención, escaneo obligatorio y control de acceso es la fuente de verdad de qué se puede desplegar. Permitir que cualquier desarrollador haga docker push directo a un registro sin control es equivalente a no tener control de cambios en absoluto.

El registro de imágenes es el punto central de control en el ecosistema de contenedores. Es donde se almacenan las imágenes que se despliegan en producción, y por lo tanto es donde se debe aplicar el control más estricto. Si el registro no tiene controles, el clúster no puede confiar en las imágenes que recibe.

Las políticas recomendadas para el registro de imágenes son:

  • Escaneo obligatorio: todas las imágenes deben ser escaneadas antes de ser aceptadas
  • Políticas de retención: las imágenes antiguas se eliminan automáticamente
  • Control de acceso: solo ciertos equipos pueden publicar imágenes en el registro
  • Tags inmutables: el registro debe permitir tags inmutables y bloquear sobrescritura de tags
  • Auditoría: registro de quién publicó qué imagen y cuándo
✔ El registro de imágenes como fuente de verdad: si la imagen no está en el registro autorizado, escaneada y con tag inmutable, no se puede desplegar en producción. El registro es el gatekeeper de qué se despliega.

Un registro bien gestionado hace que la vida del equipo de operaciones sea más fácil. En lugar de tener que verificar manualmente cada imagen que se despliega, el registro aplica las políticas automáticamente, y el clúster confía en que las imágenes que llegan del registro son seguras y conformes.
