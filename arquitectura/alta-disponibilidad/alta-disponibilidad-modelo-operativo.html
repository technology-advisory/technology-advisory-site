Modelo operativo: quién opera qué, y cuándo

El diagrama de arquitectura no salva un incidente a las 3 de la madrugada. Lo hace un runbook probado, un RACI claro y un equipo que ha ejecutado el failover antes de que le tocara hacerlo en producción bajo presión.

Miguel Ángel Carriazo Modelo Operativo Lectura Avanzada (16 min)
Módulo 01

Quién puede tocar el clúster

La mayoría de incidentes de HA que empeoran en lugar de resolverse tienen un denominador común: alguien sin el contexto completo tocó algo bajo presión. Un modelo operativo serio define con precisión quién puede iniciar un failover manual, quién puede modificar la configuración del quorum, y quién tiene autoridad para declarar "esto es un incidente, activamos el runbook".

Regla operativa: un único punto de decisión humana por incidente. Si dos personas pueden iniciar el mismo failover de forma independiente, tienes un problema de proceso, no de tecnología.
⚙ Módulo 02
Módulo 02

El runbook que sí funciona

Un runbook de failover que funciona tiene tres propiedades: está escrito para alguien que no diseñó el sistema, incluye comandos copiables, y contempla "el failover automático no ha funcionado, ¿ahora qué?"

Estructura mínima de un runbook

  • Criterio de activación: qué señal exacta dispara "esto requiere failover manual".
  • Verificación de estado previo: comandos para confirmar el estado real del quorum.
  • Secuencia de comandos exacta, con el resultado esperado de cada paso.
  • Criterio de rollback: en qué punto se aborta y se escala.

Módulo 03

Ventanas de mantenimiento vs incidente real

Un failover planificado y un failover forzado por un incidente real no son el mismo ejercicio. La prueba que realmente importa es el failover no anunciado, donde el equipo de guardia no sabe de antemano que va a ocurrir.

EscenarioQué validaQué NO valida
Failover planificadoEl mecanismo técnicoReacción bajo presión
Failover no anunciadoReacción real del equipoNada — es la prueba más completa
Incidente realTodo, mal momento para descubrirlo—
⚙ Módulo 04
Módulo 04

El failover que nadie ha probado

Un runbook sin ejecutar es una hipótesis con formato de procedimiento. La cadencia mínima defendible es un ejercicio de failover cada trimestre para sistemas críticos.

Señal de alarma: si el runbook no se ha actualizado desde que se escribió el sistema, y el sistema ha cambiado, el runbook probablemente ya no describe la realidad.
⚙ Módulo 05
Módulo 05

El post-mortem sin culpables

Cada ejecución de failover — planificada o forzada por incidente — merece un post-mortem breve y honesto. El objetivo no es encontrar a quién culpar; es que la siguiente ejecución sea mejor que esta.

Los equipos que maduran más rápido en HA son los que tratan cada failover como una oportunidad de aprendizaje, no solo los que fallan.
