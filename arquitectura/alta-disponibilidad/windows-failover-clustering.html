Windows Failover Clustering: El Detalle que Nadie Revisa

Modelos de quorum, Cluster Shared Volumes y la validación previa que evita el clúster que "funciona" hasta el primer fallo real.

Miguel Ángel Carriazo 22 Junio 2026 22 Junio 2026 Lectura Avanzada (14 min)
Módulo 01 — Validación Previa

El asistente de validación que casi nadie ejecuta bien

Windows Server incluye un Cluster Validation Wizard que comprueba red, storage y configuración antes de crear el clúster. Muchos administradores lo ejecutan, ven advertencias amarillas (no rojas) y las ignoran porque "el asistente permite continuar". Ese es exactamente el momento donde se siembra el problema que aparecerá seis meses después, en producción, sin previo aviso.

El objetivo de conmutación debe relacionarse con el RTO (Recovery Time Objective) definido para el servicio: medir únicamente si el clúster mueve el rol oculta el tiempo real hasta que la aplicación vuelve a prestar servicio de forma aceptable para negocio.

⚠️ Regla no negociable

Un clúster de producción no se despliega con ninguna advertencia de red o storage sin resolver o documentar explícitamente por qué se acepta el riesgo.

⚙ Módulo 02
Módulo 02 — Modelos de Quorum

Elegir el modelo de quorum correcto

Windows Failover Clustering soporta varios modelos de quorum: Node Majority, Node and Disk Majority, Node and File Share Majority y Cloud Witness. El error más común es usar Node Majority en un clúster de dos nodos sin ningún testigo.

Recomendación para 2 nodos: Cloud Witness si tienes conectividad a Azure. Si no, File Share Witness en un tercer servidor fuera del dominio de fallo de ambos nodos.
⚙ Módulo 03
Módulo 03 — Almacenamiento Compartido

Cluster Shared Volumes: el matiz que confunde a todos

CSV (Cluster Shared Volume) permite que múltiples nodos accedan simultáneamente al mismo volumen NTFS (New Technology File System)/ReFS. El matiz que confunde: sigue existiendo un "nodo coordinador" para las operaciones de metadatos. Si ese nodo coordinador falla, hay un breve pero real impacto en I/O.

✔ Buena práctica

Monitoriza específicamente los eventos de "CSV redirected mode" — indican que el I/O está pasando por la red del clúster en vez de la ruta directa a storage.

⚙ Módulo 04
Módulo 04 — Operación

Prioridad de arranque y Preferred Owners

Cuando un clúster completo se reinicia, el orden en que los roles y VMs arrancan importa. Un controlador de dominio virtualizado que depende de sí mismo para autenticar, o una base de datos que arranca antes que su storage, generan fallos en cascada.

⚙ Módulo 05
Módulo 05 — Monitorización

Monitorización proactiva y alertas del clúster

Un clúster correctamente configurado puede seguir funcionando durante años sin incidentes — y precisamente por eso, cuando algo empieza a ir mal, es fácil que pase desapercibido. La monitorización proactiva es la diferencia entre resolver un problema antes de que afecte a la producción y descubrirlo durante un incidente real.

Eventos críticos: ID 1135 (pérdida de nodo), ID 1146 (fallo de recurso), ID 1576 (red de clúster degradada).
