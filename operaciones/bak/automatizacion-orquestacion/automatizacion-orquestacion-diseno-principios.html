Principios de diseño: automatizar un proceso roto solo lo rompe más rápido

La automatización amplifica lo que ya existe. Si el proceso manual era confuso e inconsistente, automatizarlo tal cual produce fallos confusos e inconsistentes, pero a mayor velocidad.

1. El proceso roto, ahora ejecutándose más rápido

Automatizar un proceso manual confuso, con excepciones no documentadas y pasos que dependen del criterio de quien lo ejecuta, no arregla el proceso — lo convierte en un script que falla de forma más rápida y menos comprensible. La automatización amplifica lo que ya existe, para bien o para mal.

2. Entender el proceso real antes de automatizar, no el documentado

El proceso que realmente se ejecuta en la práctica —con sus excepciones, atajos y criterios tácitos— casi nunca coincide exactamente con el documento formal. Automatizar directamente desde el documento sin observar la ejecución real produce automatización que falla en los casos que el documento no contemplaba.

3. Manejo de errores real, no solo automatizar el camino feliz

Un script que funciona perfectamente cuando todo va bien y falla silenciosamente (o de forma catastrófica) cuando algo inesperado ocurre es automatización incompleta. El manejo explícito de errores —qué hacer cuando un recurso no existe, cuando la red falla a mitad de ejecución— es tan importante como la lógica del camino feliz.

4. Idempotencia: ejecutar el script dos veces no debería duplicar el efecto

Un script idempotente produce el mismo resultado final independientemente de cuántas veces se ejecute — crítico quando un script se relanza tras un fallo parcial, o cuando alguien lo ejecuta por error una segunda vez. Un script no idempotente que crea un recurso nuevo cada vez que se ejecuta, en lugar de verificar si ya existe, acumula duplicados silenciosamente.

5. Automatizar lo repetitivo primero, lo crítico solo con red de seguridad

Las tareas repetitivas de bajo riesgo son el punto de partida ideal para automatización — el aprendizaje se acumula con bajo coste de error. Las tareas críticas con alto impacto si fallan merecen automatización solo cuando ya existe experiencia acumulada, y siempre con capacidad de rollback verificada.
