Modelo operativo: quién prueba la restauración, y con qué frecuencia real

Un backup que no se ha restaurado nunca en un entorno de prueba es una esperanza, no una capacidad. La prueba de restauración es el control más pospuesto de toda la infraestructura.

1. El backup que nunca se restauró en dos años

Snapshots diarios ejecutándose sin fallos durante dos años, sin una sola restauración de prueba documentada, es más común de lo que debería ser. Nadie sabe si esos backups son realmente recuperables hasta el día que se necesitan de verdad — el peor momento posible para descubrirlo.

2. Verificación automática de integridad vs prueba real de restauración

La verificación automática (comprobar que el fichero de backup no está corrupto) es necesaria pero no suficiente. La prueba real de restauración —recuperar el backup en un entorno aislado y confirmar que el sistema arranca y funciona correctamente— es la única evidencia que realmente demuestra capacidad de recuperación.

3. Entorno aislado para probar sin arriesgar la producción real

Restaurar un backup en un entorno de sandbox aislado, sin conectividad a producción, permite probar la recuperación sin riesgo de interferir con sistemas activos ni de propagar datos de prueba hacia sistemas reales por error.

4. RTO real medido en cada prueba, no estimado sobre el papel

Cada prueba de restauración debería documentar el tiempo real que tardó, comparado con el RTO objetivo. La primera prueba de un sistema casi siempre revela que el tiempo real es superior al estimado — es información valiosa para ajustar expectativas antes de un incidente real, no después.

5. Cadencia defendible de pruebas según criticidad del sistema

Sistemas críticos merecen pruebas de restauración trimestrales como mínimo; sistemas de menor criticidad pueden probarse semestral o anualmente. La cadencia debería estar definida explícitamente por criticidad, no dejada a la improvisación de cuando alguien tenga tiempo.
